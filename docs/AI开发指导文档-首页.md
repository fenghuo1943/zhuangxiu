# 装修网站首页 AI 开发指导文档

## 1. 项目定位

本项目不是传统装修公司展示官网，而是面向个人或家庭装修过程的项目管理工具。首页应以“装修项目仪表盘”为核心，帮助用户快速查看预算、阶段、待办、采购、记账和流程提醒。

开发时应优先保证工具型产品的清晰、稳定和高频操作效率，不要做成营销落地页。

## 2. 首页目标

首页需要完成以下目标：

1. 让用户知道当前装修项目处于什么阶段。
2. 让用户知道今天需要处理什么。
3. 让用户快速进入预算、待办、采购、记账和流程模块。
4. 让用户看到预算是否已设置、预算余额、支出和使用率。
5. 在桌面端展示完整信息，在移动端保留最高频、最关键的信息。

## 3. 视觉风格

整体风格参考截图：

- 产品气质：温和、干净、轻量、工具型后台。
- 页面背景：浅暖灰或米白色。
- 卡片背景：白色。
- 主色：珊瑚红、橙红，用于主按钮、重要入口、强调文字。
- 辅助色：薄荷绿、浅蓝、浅橙，用于统计卡片、图标底色、状态提示。
- 字体：中文界面优先使用系统字体。
- 卡片圆角：建议 12px 到 20px，移动端可略大。
- 阴影：轻阴影，不要厚重。
- 页面留白：桌面端宽松，移动端紧凑但不能拥挤。

禁止方向：

- 不要做大面积渐变英雄区。
- 不要做营销型大横幅。
- 不要使用过度装饰。
- 不要让首页首屏变成产品介绍页。
- 不要在工具界面中堆叠大量解释性文字。

## 4. 响应式布局

### 4.1 桌面端

桌面端采用顶部导航 + 主内容区域布局。

建议断点：

- 移动端：小于 768px
- 平板端：768px 到 1023px
- 桌面端：1024px 及以上

桌面端首页结构：

1. 顶部导航栏。
2. 主内容容器，最大宽度建议 1360px 到 1440px。
3. 内容区采用左右两栏：
   - 左侧主工作区：约 70%
   - 右侧辅助区：约 30%

桌面端重点展示完整功能，包括今日提醒、今日重点、装修进度、阶段路线、预算设置、待办事项、待购清单、记账概览、常用入口和数据备份。

### 4.2 移动端

移动端采用单列流式布局。

移动端顶部不展示完整导航，只保留核心操作：

- 首页按钮
- 当前账号按钮
- 切换或返回类按钮
- 公告按钮

移动端首页内容排序建议：

1. 今日装修提醒
2. 三个摘要卡片：预算余额、待办/待购、最近支出
3. 阶段路线
4. 预算设置与阶段分配
5. 待办事项
6. 其他模块按需下移或折叠

移动端应减少同时展示的信息量，优先保证核心操作可见。

### 5.13 采购参考库 PurchaseReference

采购页面是“采购模块”的核心入口，帮助用户按装修阶段浏览参考物品并同步到首页待购清单。

#### 5.13.1 桌面端页面结构

- 标题区：图标 + 标题“采购参考库” + 简短副标题。
- 使用说明提示：说明勾选后会同步到首页待购清单。
- 统计卡片：参考物品总数、已选待购数量，以及“去首页查看待购清单”链接。
- 搜索输入：按材料名称搜索，显示搜索结果区域。
- 快速添加区域：自定义物品名称、阶段选择、数量、添加按钮。
- 操作按钮组：全部展开、全部折叠。
- 阶段树形清单：按阶段父类展开、子类折叠；每个阶段显示已选 / 总数。
- 物品行：复选框、物品名称、规格、数量输入、单位、已选/参考标签、删除按钮。
- 卡片交互：父阶段和子分组可折叠、可展开，物品可选中或取消选中。

#### 5.13.2 移动端页面结构

- 保留核心顶部导航与当前页面高亮。
- 搜索与快速添加功能必须可用且输入域宽度适配窄屏。
- 阶段树形清单在窄屏下应变成可折叠卡片列表，避免横向溢出。
- 物品行布局要适配单列，关键字段优先展示：名称、规格、数量、选中状态。
- 底部导航栏支持首页、采购、比价、记账、工具快速跳转。

#### 5.13.3 交互与功能细分

- 搜索功能：按材料名称实时过滤，结果区域单独显示。
- 勾选物品：物品可直接勾选，选中后同步到首页待购清单。
- 数量修改：允许直接编辑数量并持久化保存。
- 删除操作：支持删除参考物品，并提示确认。
- 自定义添加：用户可输入名称、选择阶段、设置数量，添加后自动选中并加入待购。
- 全部展开/全部折叠：父阶段与子分组支持批量展开或收起。
- 本地存储：数据应持久化；异常或过大时自动重置回默认数据。

#### 5.13.4 视觉与状态要求

- 采用卡片式、分层式布局，突出阶段与物品层级。
- 选中物品要有清晰视觉差异，例如背景色、边框、标签变化。
- 搜索结果和参考阶段列表要明确区分。
- 统计卡片应实时反映总数与已选数量。
- 空状态：
  - 采购库无数据时显示“暂无采购参考物品”。
  - 搜索无结果时显示“暂无符合条件的参考材料”。

#### 5.13.5 页面组件建议

- `PurchaseHeader`
- `PurchaseSummaryCard`
- `PurchaseSearch`
- `PurchaseQuickAdd`
- `PurchaseStageAccordion`
- `PurchaseSubgroup`
- `PurchaseItemRow`
- `PurchaseBottomNav`

#### 5.13.6 数据流与状态管理

- `purchaseData`：保存阶段、分组、物品数据。
- `selectedPurchaseItems`：计算已选待购条目。
- `searchQuery`：触发结果过滤。
- `expandedParents` / `expandedSubs`：控制折叠状态。

#### 5.13.7 业务说明

- 这是从“待购清单”跳转出的采购参考库页面，不是独立的物品采购频道。
- 目标是让用户快速把阶段参考采购清单转成自己项目的待购项。
- 空状态和无搜索结果状态要自然，建议显示“暂无符合条件的参考材料”。

## 5.14 记账页面（Expense / 记账）分析与实现建议

**概述**：

- 参考附件的桌面端与移动端截图与保存的 `记账.html`，页面由以下可识别区域组成：顶部账号/导航栏、总支出卡片、五类统计卡片（硬装工程/主材选购/设备系统/软装家电/服务杂项）、分类统计切换（全部账单/已支付/预付款/未支付）、导入/导出 Excel 操作、分类折叠列表（每类可展开显示子项或标签）、空状态“暂无记录”、右下角浮动新增按钮与移动端底部导航。

**从 HTML / CSS 源码中提取的要点**：

- 页面使用卡片式布局，命名空间带有 `fresh-` 前缀（如 `.fresh-project-card`、`.fresh-navbar` 等），样式采用轻柔阴影、圆角与浅暖色背景。
- 页面在窄屏（mobile）和宽屏（desktop）间通过媒体查询切换：主要行为包括隐藏/展示项目选择器、调整头像/按钮显示、将横向统计卡变为单列摘要卡。
- 页面向下滚动部分采用折叠面板（accordion）展示每个大类及其子标签（例如“硬装工程”下的合同款项、拆改、水电、瓦工等小项以 chip 形式显示）。
- 页面引入两个脚本文件 `fresh-shell.js` 与 `cloud-sync.js`（分别用于交互与云端同步/本地缓存逻辑），注意首版可以先禁用云同步，仅用本地存储作为持久化。

**布局与 UX 建议（桌面 / 移动）**：

- 桌面端：保留左侧主区域展示总览与分类折叠列表，右侧可用于快速入口或近期流水。总支出卡片放在首屏左上，统计卡按网格展示，折叠列表使用卡片阴影区分层级。
- 移动端：采用单列流式，先显示总支出 + 三个摘要卡（预算/待办/最近支出），分类统计折叠成可收起的卡片，浮动“新增”按钮固定右下且预留足够安全距离以免遮挡底部导航。
- 折叠列表项的展开/收起应有明显动画并保留状态（expanded ids）以便用户切换筛选后能保留阅读位置。

**可访问性与可交互建议**：

- 卡片与按钮应使用语义化元素（`button`, `header`, `nav`, `section`, `ul/li`）并加 `aria-expanded`、`aria-controls`、`role="tablist"/"tab"` 等以支持屏幕阅读器。
- 表单输入（金额、数量）应限制输入类型、自动千分位格式化并在输入失焦时校验避免 `NaN` 或空字符串显示为 `--` 或 `0.00`。
- 导入/导出 Excel 操作应提供明确的文件类型说明与错误提示（例如列缺失、金额格式错误），并在导入时做批量校验后给出可回滚的预览（preview + confirm）。

**前端组件拆分建议**：

- `ExpenseHeader`：顶部导航与账号按钮（可复用 `DesktopHeader`/`MobileHeader`）。
- `ExpenseSummaryCard`：总支出与小统计卡的组合（支持占比颜色、百分比）。
- `ExpenseFilters`：状态筛选（全部/已支付/预付款/未支付）、导入导出按钮组。
- `ExpenseCategoryAccordion`：分组折叠面板，包含 `ExpenseCategoryChips` 用于展示子项快速入口。
- `ExpenseListEmpty`：空状态组件（建议文案“暂无记录”并给出“记一笔”CTA）。
- `ExpenseAddModal`：新增记账弹窗/页面（支持类目、阶段、金额、日期、支付状态、备注、附件）。

**数据模型补充**：

增加或完善 `Expense` 与 `Category` 模型字段：

```ts
type Expense = {
  id: string;
  projectId: string;
  title: string; // 条目标题
  amount: number; // 单位：分或元（在后端统一）
  categoryId: string; // 对应 BudgetCategory 或自定义分类
  stageId?: string;
  date: string; // ISO 日期
  status: 'paid' | 'prepaid' | 'unpaid' | 'refunded';
  payer?: string;
  note?: string;
  createdAt: string;
};

type Category = {
  id: string;
  name: string;
  color?: string;
  order?: number;
};
```

还需在首页 mock 数据中加入记账相关字段：`expenses: Expense[]`、`expenseSummary`（按 category 和状态汇总）、`recentExpenses`。

**交互细节与业务规则**：

- 统计卡显示按月/按项目/按阶段切换（可默认“全部时间 / 当前项目”）。
- 筛选（状态/分类）要与折叠面板联动：在筛选存在结果时突出展示对应分类并自动展开；无结果显示空状态提示并给出“导入 Excel / 新增记账”操作。
- 新增/编辑/删除操作应支持撤销提示（Toast + 撤销按钮）以减少误操作损失。
- 汇总数字在大量数据时，应通过服务器分页/聚合查询；首版本地化可使用分页并在前端做聚合计算。

**移动端注意事项**：

- 底部导航与浮动新增按钮互不遮挡，浮动按钮建议距离底边至少 80px（考虑虚拟按键和底部导航高度）。
- 卡片间距与字体在 375px 宽度下要保证可读且点击目标不小于 44px。

**验收条目（补充到文档验收清单）**：

- 记账页面在桌面端展示总支出卡、分类统计卡与分类折叠列表。
- 移动端展示总支出和三个摘要卡，折叠列表可展开并显示子项 chips。
- 支持按 `已支付/预付款/未支付` 筛选并正确反映条目数量与金额。
- 导入/导出 Excel 能够触发本地校验并显示导入预览（或至少显示错误提示）。
- 新增记账能够立即刷新统计卡与最近支出列表，且不会出现 `NaN`/`undefined`。

**实现优先级建议（最小可用产品）**：

1. 静态还原：实现桌面与移动端的静态布局（折叠列表、卡片、浮动按钮）。
2. 本地交互：新增记账 Modal、按状态/分类筛选、本地聚合统计并持久化到 `localStorage`。
3. 导入/导出：实现简单的 CSV/Excel 导入导出（先 CSV 更简单），并做格式校验。
4. 云同步（可选）：接入后端 API 或云存储，优先实现只读同步与上传冲突策略。

## 11. 开发阶段规划

### 阶段一：项目基础

目标：建立可运行的前端项目。

任务：

1. 初始化前端框架。
2. 配置路由。
3. 配置基础样式。
4. 定义颜色、间距、圆角、阴影等设计变量。
5. 建立桌面端和移动端布局。

验收标准：

- 项目可以本地启动。
- 首页路由可以访问。
- 桌面和移动端布局能根据屏幕宽度切换。

### 阶段二：首页静态还原

目标：完成参考图首页的静态视觉结构。

任务：

1. 实现顶部导航。
2. 实现今日提醒。
3. 实现今日重点或移动端摘要卡片。
4. 实现装修进度。
5. 实现阶段路线。
6. 实现预算设置与阶段分配。
7. 实现待办事项。
8. 实现右侧辅助模块。

验收标准：

- 桌面端布局接近参考图。
- 移动端布局接近参考图。
- 页面没有明显重叠、溢出和错位。

### 阶段三：首页基础交互

目标：让首页具备最小可用能力。

任务：

1. 总预算输入后更新预算摘要。
2. 阶段预算输入后更新分配状态。
3. 新增待办后列表更新。
4. 完成待办后进度更新。
5. 空状态和有数据状态正常切换。

验收标准：

- 用户可以完成“设置预算”和“新增待办”两个核心动作。
- 页面金额、数量、进度显示正确。

### 阶段四：模块页面扩展

目标：根据后续参考图继续开发其他页面。

优先顺序：

1. 预算管理页
2. 待办事项页
3. 装修流程页
4. 采购清单页
5. 记账管理页
6. 工具页
7. 项目管理页

每新增一个页面，都应先分析页面结构，再复用首页已有组件。

### 阶段五：数据持久化

目标：让用户数据可以保存。

可选方案：

1. 第一版使用浏览器本地存储。
2. 后续接入后端数据库。
3. 支持导入导出 JSON 文件。

## 12. AI 编程执行规则

后续 AI 编程时应遵守：

1. 先阅读本文档，再查看当前代码结构。
2. 优先实现首页，不要提前开发未提供参考图的复杂页面。
3. 优先复用组件，不要在每个模块重复写卡片、按钮和标题样式。
4. 保持桌面端和移动端同时可用。
5. 每完成一个阶段，都要运行项目并检查页面。
6. 修改样式时要同时检查 375px 移动端和 1440px 桌面端。
7. 不要引入与装修项目管理无关的营销内容。
8. 不要把未实现功能伪装成已实现；可以先做静态入口或禁用态。
9. 所有金额显示都要有默认值，避免出现 `NaN`。
10. 所有列表都要有空状态。

## 16. 装修流程页面分析

装修流程页是项目的核心业务页面之一，用于把装修过程拆成可推进、可验收、可学习的阶段清单。页面不是普通文章列表，而是“流程推进 + 阶段提示 + 完成状态”的组合型工具页面。

页面目标：

1. 让用户知道完整装修流程有哪些步骤。
2. 让用户知道当前下一步应该做什么。
3. 让用户按阶段查看施工说明、施工标准、验收标准和攻略文章。
4. 让用户可以标记阶段完成，并自动更新整体进度。
5. 支持新房毛坯和旧房改造两套流程切换。

## 17. 装修流程页布局

### 17.1 桌面端布局

桌面端采用顶部导航 + 页面容器 + 左右两栏结构。

页面顶部为流程 Hero 区：

- 小标签：装修进程管家
- 主标题：按阶段推进，少漏项、少返工
- 说明文字：把装修流程拆成可勾选节点，每个节点保留施工标准、验收标准和攻略内容，方便做步骤执行和核对。
- 流程类型切换：新房毛坯 / 旧房改造
- 操作按钮：编辑顺序
- 统计卡片：
  - 整体进度，例如 `0 / 22`
  - 当前建议关注，例如“设计与开工准备”
  - 阶段数量，例如 `22 个阶段`

主内容区：

- 左侧：流程阶段导航卡片。
- 右侧：阶段时间线列表。

左侧阶段导航：

- 标题：流程阶段
- 按钮：全部展开、全部折叠
- 阶段 chip 列表
- 当前待处理阶段高亮
- 点击阶段后滚动到对应阶段卡片

右侧阶段列表：

- 每个阶段是一个可折叠卡片。
- 卡片头部显示编号、阶段名称、预计天数、展开按钮。
- 展开后显示阶段说明、施工标准、验收标准、攻略文章、避坑指南、完成勾选。

### 17.2 移动端布局

移动端采用单列布局。

排序建议：

1. 移动端顶部导航
2. 流程 Hero 区
3. 流程类型切换
4. 统计卡片，单列展示
5. 流程阶段导航
6. 阶段折叠列表

移动端规则：

- 顶部标题字号要缩小，避免换行过多。
- 统计卡片改为单列。
- 左侧阶段导航改为普通卡片，不使用 sticky。
- 阶段列表卡片宽度占满。
- 阶段预计天数在极窄屏可以隐藏。
- 阶段关联提示在移动端单列展示。

## 23. 比价页面（采购比价）分析与实现建议

**概述**

- 比价页面用于按品类/型号收集多个渠道报价、对比并同步到待购清单或采购流程。页面既是桌面端的管理面板，也是移动端的快速录入/对比界面。

**布局结构（从 HTML 与截图抽取）**

- 顶部：全局导航与账户入口（与首页共用 `DesktopHeader`/`MobileHeader`）。
- 操作区：标题、说明、全展/对比/线上线下切换按钮组。
- 搜索与快速添加：品类输入与“添加品类”按钮；导出/导入按钮。
- 统计卡片行：品类数、型号数、报价渠道数（卡片式 `.fresh-project-card` 风格）。
- 列表区：按品类分组的可折叠卡片（每组显示图标、品类名、型号数量、展开区为型号行）。
- 型号行：包含型号、尺寸/规格、备注、数量、以及“+ 型号”添加按钮。

**响应式规则（从源码与截图观察）**

- 宽屏（>=1024px）：使用卡片网格布局，顶部操作与统计卡并列，列表区为宽内容区并居中（最大宽度约 1360px~1440px）。
- 中宽（680px~1023px）：导航压缩，部分按钮长度收缩，卡片与表单项保持两列或自适应列。
- 窄屏（<680px）：导航只显示关键元素（项目选择器、公告、账号），`.fresh-project-nav` 隐藏，使用`.fresh-project-toggle` 显示项目弹出；列表卡片转为单列，操作按钮与输入做堆叠；底部保留导航栏快速跳转（首页/采购/比价/记账/工具）。

**关键组件建议**

- `PriceHeader`：标题、说明、切换按钮组（线上/线下/比较模式）、导入导出。
- `PriceSearchAdd`：品类搜索 + 快速添加品类控件（支持占位示例“如：冰箱、瓷砖、木门”）。
- `PriceStats`：三卡统计（品类/型号/报价渠道），支持小屏下折叠为可滑动卡片。
- `PriceCategoryCard`：品类折叠卡，头部显示图标、品类名、型号数量、删除与展开操作；主体为型号行列表。
- `PriceModelRow`：型号录入行，字段：型号、尺寸/规格、备注、数量、渠道价格引用/同步按钮。
- `PriceFooterActions`：新增型号按钮、批量导入、同步到待购清单。

**交互细节与业务规则**

- 新增品类：输入文本后点击“添加品类”创建折叠卡（默认展开或按用户偏好记忆展开状态）。
- 新增型号：在对应品类下添加型号行；每行保存本地并可导出为 CSV/Excel。
- 报价渠道：每个型号应支持多个渠道报价记录（结构化为数组），可在对比模式展开对比表格并计算最低价/平均价。
- 导入/导出：支持 CSV 优先，字段校验（必填：品类、型号；可选：规格、备注、渠道、价格）。导入失败需给出行级错误并可回滚。
- 同步：用户勾选型号后，可将选中条目同步到首页的待购清单（`selectedPurchaseItems`），并在同步后更新统计卡。

**数据模型（建议）**

```ts
type PriceCategory = {
  id: string;
  name: string;
  icon?: string;
  models: PriceModel[];
};

type PriceModel = {
  id: string;
  name: string;
  spec?: string;
  note?: string;
  quantity?: number;
  channelQuotes: ChannelQuote[]; // 多渠道报价
};

type ChannelQuote = {
  id: string;
  channel: string; // 淘宝/京东/线下/供应商
  price?: number;
  url?: string;
  updatedAt?: string;
};
```

**无障碍与可访问性**

- 折叠面板使用 `button` 并设定 `aria-expanded`、`aria-controls`。
- 表单输入标注 `label`；错误提示与导入结果使用 `role="alert"`。 
- 操作按钮保证触控目标至少 44x44px（移动端）。

**样式与实现注意（根据源码片段）**

- 现有源码使用命名空间 `fresh-`（例如 `.fresh-project-card`、`.fresh-project-nav`），可复用这些类或沿用 BEM 命名一致性。
- 卡片圆角、轻阴影、浅暖色背景与主色 `#e45b3f`（珊瑚红）需与全局变量一致。
- 在窄屏下使用 `position: fixed` 的底部导航，并为浮动操作留出安全距离（至少 80px）。

**开发优先级建议（比价模块）**

1. 静态页面还原：实现桌面与移动端静态布局与折叠交互（不接后端）。
2. 本地持久化：新增品类/型号、导入/导出 CSV、本地保存（`localStorage`）。
3. 多渠道报价对比：模型支持多渠道报价并展示最低价标签。 
4. 同步到待购：实现从比价页面到首页待购清单的同步动作与统计更新。

**验收清单（比价模块）**

- 桌面端展示标题区域、切换按钮、统计卡、按品类分组的折叠列表。
- 移动端展示简化搜索/添加、统计卡为摘要卡、折叠卡单列且可滚动。
- 可新增品类与型号并保存到本地（刷新后能恢复）。
- 导入 CSV 并报告错误行；导入成功能将条目插入到正确的品类/型号下。
- 勾选型号后能同步到首页待购清单并更新统计卡。

**参考实现提示**

- 可以先用纯前端框架（React/Vue）快速实现组件并用 mock 数据驱动。优先做可复用的 `PriceCategoryCard` 与 `PriceModelRow` 组件。
- 导入/导出建议采用 CSV 作为首选格式，后续支持 Excel（xlsx）导入。

- 若后续需要正文详情，再单独设计资料详情页或弹窗。
- 如果某阶段没有关联资料，显示空状态：该节点暂未关联扩展资料，请以施工与验收要点为准。

## 23. 装修流程页交互与验收

核心交互：

1. 新房毛坯 / 旧房改造切换。

6. 自动计算整体进度。
7. 自动定位当前建议关注阶段，即第一个未完成阶段。
8. 编辑流程顺序，保存后保持完成状态映射。

const FLOW_PROGRESS_KEY = 'renovation_flow_progress';

- 桌面端顶部导航中“流程”处于激活态。
